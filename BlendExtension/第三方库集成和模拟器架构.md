## iOS 第三方库集成方式：从 2015 到现在的演变

---

### 一、你项目中用到的三种集成方式

#### 方式1：手动拖入 .a 静态库（最原始）

**项目实例：** `08-01-测试.a静态库`、友盟分享（`libUMSocial_Sdk_4.4.a`）、极光推送（`libPushSDK-1.8.8.a`）

```
需要给使用者三样东西：
├── libXxx.a          ← 二进制库（编译后的代码）
├── include/Xxx.h     ← 头文件（告诉你有哪些方法）
└── Xxx.bundle        ← 资源文件（图片等）
```

```objc
#import "HMTool.h"   // 引号导入
```

**缺点：**
- 头文件、库文件、资源分散，容易漏
- 需要手动配 Header Search Path、Library Search Path
- 升级版本要手动替换所有文件
- 架构问题需要自己合并（`lipo -create`）

---

#### 方式2：手动拖入 .framework（进步了一点）

**项目实例：** `08-03-测试Framework`、百度地图（`BaiduMapAPI_xxx.framework`）、支付宝（`AlipaySDK.framework`）

```
只需要给一个文件夹：
└── HMFramework.framework/
    ├── HMFramework         ← 二进制库
    ├── Headers/            ← 头文件
    ├── Modules/            ← 模块信息
    └── Info.plist          ← 版本信息
```

```objc
#import <HMFramework/HMFramework.h>  // 尖括号 + 模块名导入
```

**比 .a 好在：** 一个文件夹包含一切，拖进去就能用
**仍然的缺点：** 升级要手动替换、架构问题要自己处理

---

#### 方式3：CocoaPods（自动化管理）

**项目实例：** 网易新闻（`AFNetworking`）

```ruby
# Podfile 声明依赖
target '01-网易新闻' do
    pod "AFNetworking"
end
```

```bash
pod install  # 一条命令搞定一切
```

**自动完成的事：**
- ✅ 下载源码/库文件
- ✅ 配置 Header Search Path
- ✅ 配置 Link 选项
- ✅ 处理依赖关系（A 依赖 B，自动装 B）
- ✅ 升级只需改版本号 + `pod update`

---

### 二、三种方式对比

| | 手动 .a | 手动 .framework | CocoaPods |
|---|---|---|---|
| 文件结构 | 散装 | 打包好的文件夹 | 自动管理 |
| 配置难度 | 高（手动配路径） | 中（拖入即可） | 低（一条命令） |
| 升级方式 | 手动替换 | 手动替换 | `pod update` |
| 处理依赖 | 手动 | 手动 | 自动 |
| 架构问题 | 手动 `lipo` 合并 | 手动处理 | 自动处理 |
| 源码可见 | ❌ | ❌ | ✅（大部分 Pod 是源码） |
| 你项目的例子 | 友盟、极光 | 百度地图、支付宝 | AFNetworking |
| 2015 年的用法 | ⭐ 非常常见 | ⭐ 非常常见 | 开始流行 |
| 现在的用法 | 很少用 | 偶尔用 | ⭐ 主流 |

---

### 三、现在（2025年）的主流做法

#### 1. CocoaPods — 仍然是 OC 项目的主流

```ruby
# Podfile
platform :ios, '15.0'

target 'MyApp' do
    pod 'AFNetworking', '~> 4.0'
    pod 'SDWebImage', '~> 5.0'
    pod 'MBProgressHUD'
end
```

```bash
pod install
```

#### 2. Swift Package Manager（SPM）— Swift 项目的首选

苹果亲儿子，Xcode 内置，不需要额外安装任何工具：

```
Xcode → File → Add Package Dependencies → 输入 GitHub 地址 → 自动集成
```

不需要 Podfile，不需要 Workspace，不需要命令行。

#### 3. Carthage — 小众但还在用

```bash
# Cartfile
github "Alamofire/Alamofire"

carthage update  # 编译成 .framework
```

编译出 `.framework` 后手动拖入项目，介于手动和全自动之间。

---

### 四、你项目中"跑不了"的项目都是什么原因

| 项目 | 集成方式 | 跑不了的原因 |
|------|---------|-------------|
| 百度地图 | 手动 .framework | SDK 太老 + 缺 arm64-sim 架构 + `libstdc++` 被移除 |
| 友盟分享 | 手动 .a | SDK 太老 + 缺 arm64-sim 架构 |
| 支付宝支付 | 手动 .framework + .a | SDK 太老 + 缺 arm64-sim 架构 |
| 极光推送 | 手动 .a | SDK 太老 + 缺 arm64-sim 架构 |
| iAd 广告 | 系统框架 | 苹果 2020 年彻底移除了 iAd |
| 网易新闻 | CocoaPods | ✅ Podfile 修一下就跑起来了 |

**规律：** 手动集成的全部跑不了，CocoaPods 管理的改一下就好了。

**根本原因：** 手动集成的 `.a` / `.framework` 是 2015 年编译的二进制文件，只包含当时的 CPU 架构（`armv7`、`i386`、`x86_64`、`arm64`），没有 Apple Silicon Mac 模拟器需要的 `arm64-sim` 架构。而 CocoaPods 管理的库在 `pod install` 时会重新拉取最新源码编译，自然支持新架构。

---

### 五、如果要让这些老项目跑起来

```
方案1（推荐）：换用 CocoaPods 重新集成
  → 去掉手动拖入的 .a/.framework
  → 写 Podfile，pod install
  → 修改 import 语句

方案2：去官网下载最新 SDK
  → 百度地图：https://lbsyun.baidu.com
  → 友盟：https://www.umeng.com
  → 支付宝：https://open.alipay.com
  → 极光：https://www.jiguang.cn
  → 按新文档重新集成

方案3：只在真机上运行
  → 绕过模拟器架构问题
  → 但 SDK API 可能还是不兼容
```

---

### 六、一图总结

```
2015 年                          2025 年
─────────                       ─────────
手动拖 .a        ───淘汰──→      CocoaPods（OC 主流）
手动拖 .framework ───淘汰──→     SPM（Swift 主流）
CocoaPods 刚流行  ───成熟──→     CocoaPods（稳定好用）
                                Carthage（小众）

你的项目：
百度地图/友盟/支付宝/极光  →  手动集成  →  ❌ 跑不了
网易新闻 AFNetworking     →  CocoaPods →  ✅ 改一下就好
```

**总结一句话：能用包管理工具就别手动拖，CocoaPods/SPM 才是正确的姿势。**

## 现在的 iOS CPU 架构

---

### 一、2015 年的架构（你项目里注释写的）

```
模拟器：
  i386    → 32位模拟器（iPhone 4S/5）
  x86_64  → 64位模拟器（iPhone 5S~6S）

真机：
  armv7   → iPhone 4/4S
  armv7s  → iPhone 5/5C
  arm64   → iPhone 5S~6S
```

那时候一共 **5 种**架构，开发要合并来合并去很痛苦。

---

### 二、现在（2025 年）的架构

#### 真机

```
arm64   ← 唯一的真机架构
```

从 iPhone 5S 到 iPhone 16，全部都是 `arm64`。苹果已经淘汰了 32 位设备（`armv7`、`armv7s`），Xcode 也不再支持编译这些架构。

#### 模拟器

```
Intel Mac 模拟器  → x86_64
Apple Silicon Mac 模拟器 → arm64（arm64-sim）
```

注意：Apple Silicon Mac（M1/M2/M3/M4）上的模拟器也是 `arm64`，但和真机的 `arm64` **不一样**，是带模拟器标记的 `arm64-sim`。

---

### 三、对比变化

| | 2015 年 | 2025 年 |
|---|---|---|
| 真机 32 位 | `armv7`、`armv7s` | ❌ 淘汰 |
| 真机 64 位 | `arm64` | `arm64`（唯一） |
| 模拟器 32 位 | `i386` | ❌ 淘汰 |
| 模拟器 64 位 | `x86_64` | `x86_64`（Intel Mac） |
| 模拟器 ARM | 不存在 | `arm64-sim`（Apple Silicon Mac）|
| 总共 | 5 种 | 2~3 种 |

---

### 四、为什么你的老项目跑不了

```
2015 年编译的静态库包含：
  armv7 ✅  armv7s ✅  i386 ✅  x86_64 ✅  arm64 ✅

你现在的 Mac（Apple Silicon）模拟器需要：
  arm64-sim ← 2015 年根本不存在这个架构！
```

所以报错：`building for 'iOS-simulator', but linking in object file built for 'iOS'`

老库里的 `arm64` 是真机的，不是模拟器的，编译器分不清。

---

### 五、现在开发需要关心的

```
开发时（模拟器）：
  Intel Mac    → x86_64
  M 系列 Mac   → arm64-sim

发布时（真机）：
  arm64        ← 全部 iPhone 都是这一个
```

现在架构变得非常简单了：

| 你的 Mac | 模拟器架构 | 真机架构 |
|----------|-----------|---------|
| Intel Mac | `x86_64` | `arm64` |
| M1/M2/M3/M4 Mac | `arm64-sim` | `arm64` |

用 CocoaPods 或 SPM 集成的库完全不用操心这些，包管理工具会自动处理。只有手动拖入的老 `.a` / `.framework` 才会遇到架构问题。
