### 一、项目中的问题：循环引用

```objc
@property (nonatomic, strong) Dog *dog;       // Person 强持有 Dog
@property (nonatomic, strong) Person *person;  // Dog 强持有 Person
```

```objc
- (void)touchesBegan:(NSSet<UITouch *> *)touches withEvent:(UIEvent *)event {
    Person *p = [Person new];   // p 引用计数 = 1
    Dog *d = [Dog new];         // d 引用计数 = 1
    
    p.dog = d;       // d 引用计数 = 2
    d.person = p;    // p 引用计数 = 2
}
// 方法结束：p 和 d 局部变量销毁
// p 引用计数 2→1，d 引用计数 2→1
// 都不为 0 → 永远释放不了 → 内存泄漏！
// dealloc 里的 "人死了" "狗死了" 永远不会打印
```

```
  Person ──strong──→ Dog
    ↑                  │
    └───── strong ─────┘
       💀 死锁，谁也释放不了谁
```

### 二、三种内存分析方式

#### 1. 静态分析（Analyze）— 编译时检查

| 项目 | 说明 |
|------|------|
| 快捷键 | `Shift + Cmd + B` |
| 原理 | **不运行程序**，编译器扫描代码逻辑 |
| 能发现 | 未释放的 `malloc`/`CF` 对象、逻辑上可能的泄漏 |
| 发现不了 | **循环引用**、运行时才出现的问题 |

适合做第一轮快速排查，但能力有限。

#### 2. 动态分析（Instruments - Leaks）— 运行时检测 ⭐

| 项目 | 说明 |
|------|------|
| 快捷键 | `Cmd + I` |
| 原理 | **真正运行程序**，实时追踪每个对象的内存分配和释放 |
| 能发现 | 循环引用、任何运行时的内存泄漏 |
| 界面 | 两条轨道：**Allocations**（分配）+ **Leaks**（泄漏） |

**使用步骤：**

```
1️⃣  Cmd + I → 选择 Leaks 模板 → 录制
2️⃣  操作 App（触发泄漏场景）
3️⃣  停止录制，查看 Leaks 轨道
4️⃣  红色 ✕ = 有泄漏
5️⃣  下方列表显示泄漏的对象和大小
6️⃣  切换到 Call Tree → 勾选 Hide System Libraries
7️⃣  只显示你自己代码中的泄漏
8️⃣  双击进入源码，精确定位到哪一行
```

从你的截图中可以看到：
- **Leaks 列表**：Dog 泄漏 3 次 48 Bytes，Person 泄漏 3 次 48 Bytes（你点了 3 次屏幕）
- **Call Tree**：定位到 `-[ViewController touchesBegan:withEvent:]`
- **源码定位**：精确到第 24、25 行的 `[Person new]` 和 `[Dog new]`

#### 3. Debug Memory Graph — Xcode 内置可视化 ⭐⭐

这是 Xcode 8 之后新增的工具，比 Instruments 更直观：

| 项目 | 说明 |
|------|------|
| 使用方式 | 运行时点击 Debug 栏的 **🔲** 图标（Debug Memory Graph） |
| 原理 | 暂停程序，生成所有对象的引用关系图 |
| 优势 | **可视化看到循环引用的箭头**，一目了然 |

### 三、ARC 下常见的内存泄漏场景

ARC 帮你管理了大部分内存，但以下场景仍然会泄漏：

#### 场景 1：循环引用（strong - strong）

就是这个项目演示的情况：

```objc
// ❌ 泄漏
@property (nonatomic, strong) Dog *dog;
@property (nonatomic, strong) Person *person;

// ✅ 修复：一方改为 weak
@property (nonatomic, strong) Dog *dog;
@property (nonatomic, weak) Person *person;
```

#### 场景 2：Block 循环引用

```objc
// ❌ 泄漏：self 持有 block，block 捕获了 self
self.completionBlock = ^{
    [self doSomething];  // block 强引用 self
};

// ✅ 修复：__weak 打破循环
__weak typeof(self) weakSelf = self;
self.completionBlock = ^{
    [weakSelf doSomething];
};
```

#### 场景 3：NSTimer 循环引用

```objc
// ❌ 泄漏：Timer 强引用 target(self)，self 又持有 Timer
self.timer = [NSTimer scheduledTimerWithTimeInterval:1.0
                                              target:self    // Timer → self（strong）
                                            selector:@selector(tick)
                                            userInfo:nil
                                             repeats:YES];

// ✅ 修复：在 viewWillDisappear 中 invalidate
- (void)viewWillDisappear:(BOOL)animated {
    [self.timer invalidate];
    self.timer = nil;
}
```

#### 场景 4：delegate 循环引用

```objc
// ❌ 泄漏
@property (nonatomic, strong) id<MyDelegate> delegate;

// ✅ 修复：delegate 用 weak
@property (nonatomic, weak) id<MyDelegate> delegate;
```

#### 场景 5：Core Foundation 对象

```objc
// ARC 不管理 CF 对象，必须手动释放
CGImageRef imageRef = CGImageCreate(...);
// ... 使用
CGImageRelease(imageRef);  // 必须手动释放
```

### 四、工具对比

| | 静态分析 | Instruments Leaks | Debug Memory Graph |
|---|---|---|---|
| 快捷键 | `Shift+Cmd+B` | `Cmd+I` | 运行时点图标 |
| 是否运行程序 | ❌ 不运行 | ✅ 运行 | ✅ 运行 |
| 循环引用 | ❌ 发现不了 | ✅ 能发现 | ✅ 能发现且可视化 |
| Block 泄漏 | ❌ | ✅ | ✅ |
| CF 对象泄漏 | ✅ | ✅ | ✅ |
| 定位到代码行 | ✅ | ✅ | ✅ |
| 可视化引用关系 | ❌ | ❌ | ✅ 箭头图 |

### 五、最佳实践

```
开发阶段
  ├── Shift+Cmd+B 静态分析 → 快速排查明显问题
  ├── Debug Memory Graph → 可视化查看引用关系
  └── Cmd+I Instruments → 深度检测运行时泄漏

编码规范
  ├── delegate 用 weak
  ├── Block 中用 __weak/__strong
  ├── NSTimer 及时 invalidate
  ├── CF 对象手动 Release
  └── 注意 strong-strong 循环引用

测试建议
  ├── 用真机测试（模拟器内存数据不准）
  └── 用 Release 模式（更接近上线环境）
```
